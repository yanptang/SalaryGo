# M6–M10 实现说明

M6 同时执行组合级 `single_instrument` 上限和候选产品级 `maximum_weight` 上限。个股候选必须使用用户档案中的单只个股上限，不能只依赖个股桶合计上限。

## M6 资金分配与风险校验

`allocation-v1` 只分配新增资金，按每种投入币种分别保证金额守恒。资金优先修正目标资金桶缺口，并受单一标的、个股合计、市场、币种、目标上限和最小交易金额约束。候选全部不合格时保留现金；数据过期、缺汇率或硬风控失败时返回 `blocked` 且正式建议为空。普通水温变化不触发卖出。

## M7 Agent 与策略记录

Agent 提取明确的金额和币种，信息含糊时返回追问。确定性引擎通过后才允许保存策略 JSON 和 Markdown 报告。建议、执行状态和实际成交分开保存。根目录 `AGENTS.md` 定义 Codex 的持久工作边界；Codex 不共享登录态给页面，也不自行计算金额。

## M8 本地页面

统一入口覆盖总览、持仓、策略、历史、候选池、回测和设置。数据来自本地 Python 服务，页面刷新不丢失。服务固定绑定 `127.0.0.1`，校验 Host、Origin 和随机 CSRF token，并设置浏览器安全响应头。

页面包含资金桶、账户、币种、历史市值、水温分项及策略前后配置。持仓可新增和编辑；结构化设置通过页面校验保存，无需打开底层私有文件。

## M9 回测

`backtest-v1` 比较固定定投、固定配置、水温倾斜和水温加防守四种规则。报告包含费用前后、资金加权年化、波动、最大回撤、回撤持续时间、最差月份、滚动收益、信号审计及相对固定定投差异。只有 `published_at <= 投资日` 的信号可用。

## M10 闭环

应用服务串联档案、三类账户持仓、市场刷新、人民币/瑞典克朗请求、结构化建议、报告、实际成交和恢复读取。生成建议不会修改持仓；只有用户明确提交实际成交后，账本和执行状态才更新。

## 当前数据源边界

代码已实现统一行情 provider、缓存、来源、时间、失败和过期状态。公开仓库内只有非个人 fixture，页面的“刷新测试数据”明确不能用于真实投资。正式使用前必须在页面手动录入最新价格/净值，或后续接入经确认的生产数据源；系统不得静默把 fixture 升格为真实数据。
