# SalaryGo 第一版 MVP 需求文档

> 文档版本：1.0-rc1  
> 状态：待用户确认  
> 产品范围：单用户、本地运行、仅供个人使用  
> 目标设备：Apple Silicon Mac（当前为 M1 Mac）  
> 更新日期：2026-07-18

## 1. 产品定义

SalaryGo 是一个本地运行的个人投资决策助手。用户通过 Agent 完成个人信息初始化、持仓维护和投资目标设置；当用户有新增可投资资金时，Agent 读取最新持仓和市场数据，生成可解释、可记录的投资策略及本次资金分配建议。

产品不自动连接券商或执行交易。用户在东方财富或 Revolut 手动完成交易后，再更新实际持仓。

### 1.1 核心价值

用户只需要表达：

```text
这个月有 15,000 元人民币可以投资。
Revolut 还有 5,000 瑞典克朗可以投资。
```

SalaryGo 应回答：

- 当前组合处于什么状态；
- 本次资金应该投向什么；
- 每项投入多少；
- 哪部分资金暂时保留；
- 为什么这样分配；
- 使用了哪些数据和规则；
- 主要风险和判断失效条件是什么；
- 是否需要对已有持仓进行操作。

## 2. MVP 成功标准

第一版完成后，用户能够独立完成以下闭环：

1. 与 Agent 进行多轮对话，完成个人信息和投资约束初始化。
2. Agent 将初始化结果保存为本地结构化记录。
3. 用户录入并维护东方财富、港股通和 Revolut 的实际持仓。
4. 用户手动刷新市场数据和基金净值。
5. 用户输入本次新增可投资资金。
6. 系统基于持仓、市场数据和风险规则生成投资策略。
7. 用户能够查看具体金额、资产流向、数据依据、风险和备选方案。
8. 系统将策略保存为机器可读记录和人类可读报告。
9. 用户可以在本地页面查看当前持仓、净值、组合结构和历史策略。
10. 各功能模块能够独立测试，再完成端到端测试。

## 3. 用户与运行边界

### 3.1 用户范围

- 仅支持一个本地用户。
- 不实现注册、登录、密码找回、多人协作和用户切换。
- 不向家人、朋友或公众提供在线服务。

### 3.2 已知账户

- 东方财富证券账户：支持 A 股和港股通。
- Revolut 账户：可以收到瑞典克朗，并购买账户内实际可见的美股或 ETF。
- 具体交易权限、产品可用性、费用和限额以账户实际页面为准。

### 3.3 运行方式

- Agent 优先复用用户当前登录的 Codex。
- 确定性计算和个人数据保存在本机。
- 提供一个本地可交互页面作为统一入口和数据看板。
- 前端可以是静态构建产物，但持仓写入、数据刷新、Agent 结果读取和本地文件保存需要轻量本地服务支持，因此 MVP 不定义为纯静态网页。
- 页面仅绑定本机地址，默认不开放局域网和公网访问。

## 4. 投资范围

### 4.1 四个资金桶

1. **现金管理桶**
   - 货币基金或类似的低波动现金管理产品；
   - 用于保存暂未投入的资金；
   - 不等同于银行存款，不承诺保本。

2. **纯债防守桶**
   - MVP 优先考虑短债或中短债高质量纯债基金；
   - 用于降低组合波动；
   - 仍需提示利率、信用、久期和流动性风险。

3. **指数核心桶**
   - 中国宽基指数基金；
   - 美国宽基指数基金或 ETF；
   - 纳斯达克 100 指数基金或 ETF；
   - 作为长期增长的主要来源。

4. **个股增强桶**
   - 仅包含用户人工添加的 A 股、港股和美股观察名单；
   - 不扫描全市场个股；
   - 设置单只个股和个股合计仓位上限。

### 4.2 明确排除

- 加密货币；
- 黄金类基金；
- 杠杆基金和反向基金；
- 可转债基金、二级债基和复杂固收增强产品；
- 高收益债、低评级信用债和复杂杠杆债券策略；
- 自动扫描或推荐银行理财；
- 用户未加入观察名单的个股。

## 5. 产品核心流程

### 5.1 首次初始化

Agent 通过多轮对话收集：

- 目标投资期限；
- 目标年化收益率；
- 最大可接受回撤；
- 每月通常可投资金额；
- 应急资金和未来资金用途；
- 市场大幅下跌时的预期行为；
- 当前账户和可购买市场；
- 当前持仓；
- 个股、行业、市场和币种的集中度上限；
- 是否允许部分资金暂留现金；
- 不希望投资的行业、公司或产品；
- 人民币、瑞典克朗和其他外币资金情况。

当回答存在冲突时，Agent 必须继续追问。例如，高收益目标与低回撤要求不兼容时，不能直接生成高风险策略。

初始化完成后，页面展示结构化结果，用户确认后写入本地档案。

### 5.2 持仓维护

用户可以手动新增、修改和删除持仓记录。每条持仓至少包含：

- 账户；
- 市场；
- 标的名称和代码；
- 资产类型和所属资金桶；
- 持有数量；
- 平均成本；
- 成本币种；
- 当前价格或净值；
- 当前市值；
- 数据更新时间；
- 用户备注。

MVP 不要求自动同步券商。实际成交后由用户手动更新。

### 5.3 手动刷新数据

用户点击“刷新数据”后，系统：

1. 更新候选池和持仓标的价格或净值；
2. 更新必要汇率；
3. 更新指数估值、趋势、波动和回撤数据；
4. 更新基金申购、限购、费率和折溢价等可获得状态；
5. 显示每项数据的来源、时间和刷新结果；
6. 对失败、缺失和过期数据明确提示，不静默使用旧值。

MVP 不要求实时行情。日频或用户使用时刷新即可。

### 5.4 生成投资策略

用户输入本次新增资金及币种后，系统按以下顺序处理：

1. 读取用户档案和风险约束；
2. 读取当前持仓和账户分布；
3. 检查市场数据新鲜度；
4. 计算各资金桶当前比例和目标缺口；
5. 计算指数水温；
6. 检查观察名单个股的经营和风险状态；
7. 生成新增资金候选分配；
8. 检查集中度、现金、币种和交易约束；
9. 生成最终策略和解释；
10. 保存策略记录。

建议项数不设硬上限，但必须遵循“只保留必要操作”的原则。系统不得为了覆盖所有资金桶而强行生成小额交易。

### 5.5 执行与反馈

- 系统只生成建议，不自动下单。
- 用户可以将建议标记为“计划执行”“部分执行”“已执行”或“不执行”。
- 用户手动录入实际成交数量、价格、费用和时间。
- 实际成交与原建议分别保存，便于后续复盘。

## 6. 策略与决策规则

### 6.1 基础原则

- 先根据用户风险和期限形成长期目标配置区间。
- 优先使用新增资金修正低配和超配，不因普通市场波动频繁赎回。
- 市场水温只对新增资金做有限度倾斜，不直接预测未来涨跌。
- 当候选资产不满足条件时，允许资金进入现金管理桶。
- 收益目标是评估基准，不是收益保证。
- 所有精确金额必须由确定性程序计算，模型不得自由编造。

### 6.2 指数水温

指数水温至少包含：

- 当前价格在 1 年、3 年和 5 年区间的位置；
- 距离历史高点的回撤；
- 估值相对自身历史的位置；
- 6 个月和 12 个月趋势；
- 波动率和下跌速度；
- 当前组合是否低配；
- 数据完整性和新鲜度。

水温总分必须保留子项分数，方便页面解释。

### 6.3 个股检查

人工观察名单中的个股至少检查：

- 收入和利润变化；
- 经营现金流和自由现金流；
- 盈利能力和稳定性；
- 负债及偿债能力；
- 估值相对自身历史和同行的位置；
- 价格趋势和波动；
- 审计、退市、重大违规和持续经营等红线；
- 数据发布时间和来源。

经营数据缺失或过期时，系统降低置信度或不推荐。

### 6.4 卖出和赎回

普通水温变化默认不触发卖出。可考虑卖出或赎回的情形包括：

- 个股投资逻辑失效或触发经营红线；
- 单一标的或资产桶明显超过上限；
- 产品长期限购、清盘、合并或跟踪明显恶化；
- 场内产品持续出现异常折溢价或流动性问题；
- 用户资金用途和投资期限发生变化；
- 定期再平衡达到预设阈值。

卖出门槛应高于买入门槛，并明确说明触发依据和误判风险。

## 7. 候选池与基金选择

### 7.1 最小候选结构

- 1 个现金管理主产品，最多 1 个备用产品；
- 1 个纯债主产品，最多 1 个备用产品；
- 1 个中国宽基指数，每个指数 1 个主基金和最多 1 个备用基金；
- 1 个美国宽基指数，每个指数 1 个主基金和最多 1 个备用基金；
- 1 个纳斯达克 100 指数，每个指数 1 个主基金和最多 1 个备用基金；
- 用户人工添加的少量个股观察名单。

### 7.2 选择顺序

```text
先确定指数暴露
→ 收集账户实际可购买的基金
→ 执行硬性排除
→ 比较实施质量
→ 选出主产品和备用产品
```

硬性排除条件包括：

- 当前不可购买或限额不足；
- 跟踪目标不一致；
- 杠杆、反向或结构复杂；
- 规模、流动性或存续稳定性明显不足；
- 关键费用、净值、折溢价或跟踪数据缺失；
- 场内 QDII ETF 折溢价超过规则阈值。

评分维度包括：

- 跟踪偏离和跟踪误差；
- 总持有成本；
- 规模、成交量、价差和产品历史；
- 申购、定投、赎回和限购状态；
- 场内折溢价；
- 场外确认和到账时间；
- A、C 等份额在预计持有期限下的总成本；
- 币种、换汇和操作复杂度。

### 7.3 候选池维护

- 每次生成建议前检查主产品的可购买状态、折溢价和数据新鲜度。
- 每季度正式复核一次主产品和备用产品。
- 清盘、合并、长期限购、费率明显变化、跟踪恶化或流动性风险触发即时复核。
- 出现更好的产品时，优先将后续新增资金改投新产品，不因小幅优势立即赎回旧持仓。
- 每次变更保存候选池版本、数据日期、变更理由和被替代产品。

## 8. 购买渠道规则

### 8.1 东方财富

优先承载：

- 人民币现金管理；
- 纯债基金；
- 中国宽基基金；
- A 股和港股通观察名单个股；
- 需要使用人民币配置海外指数时的场外 QDII 指数基金。

### 8.2 Revolut

优先承载：

- 收到的瑞典克朗资金；
- 账户内实际可购买的美国宽基或纳斯达克 100 ETF；
- 用户手动加入的少量美股观察名单。

### 8.3 跨渠道原则

- 同一个指数原则上只指定一个主渠道和一个备用渠道。
- 不为了形式上的分散在多个平台重复购买相同指数。
- MVP 不处理 Revolut 资金汇回国内。
- MVP 保存币种、汇率、费用和对账信息，但不提供税务或跨境合规结论。

## 9. Agent 模块

### 9.1 Agent 负责

- 多轮初始化对话；
- 理解用户本次投资请求；
- 发现信息冲突并追问；
- 调用档案、持仓、市场、评分、分配和风控工具；
- 阅读观察名单个股的可信非结构化材料；
- 解释策略和风险；
- 生成结构化输出和人类可读报告；
- 回答情景问题。

### 9.2 Agent 不负责

- 编造行情、财务、估值和汇率；
- 自由决定精确金额；
- 绕过仓位和风险限制；
- 直接修改实际持仓而不经用户确认；
- 自动交易、转账或换汇；
- 在数据过期或缺失时生成确定性建议。

### 9.3 Codex 集成

- MVP 优先由当前 Codex 作为 Agent 交互入口。
- SalaryGo 提供项目级工作流说明和本地确定性工具。
- 技术实现阶段首先验证 Codex 与本地服务之间的工具调用和记录落盘链路。
- 如果无法在本地页面中直接复用当前 Codex 会话，MVP 允许“Codex 负责对话、本地页面负责查看和编辑数据”，两者通过同一组本地记录文件同步。

## 10. 策略输出与记录

### 10.1 页面摘要

每次策略至少展示：

- 本次可投资金额及币种；
- 各资金桶当前比例、目标区间和操作后比例；
- 建议操作数量和涉及平台；
- 建议投入、保留现金和卖出金额；
- 是否存在数据异常或风险阻断。

### 10.2 单项建议详情

- 标的名称和代码；
- 购买或赎回平台；
- 操作类型；
- 建议金额、数量或比例；
- 当前持仓和操作后持仓；
- 水温总分和子项；
- 估值历史分位；
- 回撤、趋势和波动；
- 费率、跟踪误差和流动性；
- QDII 折溢价及限购状态；
- 汇率影响；
- 数据来源和更新时间；
- 推荐理由；
- 主要风险；
- 判断失效条件；
- 备用产品及未选择原因；
- 置信度和数据质量。

### 10.3 本地记录

每次策略保存两份内容：

1. 结构化记录：供程序、页面和后续复盘读取；
2. Markdown 报告：供用户直接阅读。

记录至少包含：

- 唯一策略编号；
- 生成时间；
- 用户输入；
- 使用的档案、持仓和候选池版本；
- 市场数据快照时间；
- 规则和评分版本；
- 最终建议；
- 风险校验结果；
- Agent 解释；
- 用户执行状态；
- 实际成交记录。

## 11. 本地可交互页面

### 11.1 页面范围

1. **首页/组合总览**
   - 总资产；
   - 各账户市值；
   - 各资金桶比例；
   - 币种分布；
   - 最新净值时间；
   - 最近一次策略摘要；
   - 数据异常提示。

2. **持仓页面**
   - 持仓列表；
   - 新增、编辑和删除；
   - 成本、市值、盈亏和更新时间；
   - 按账户、市场、资金桶和币种筛选。

3. **策略页面**
   - 输入本次可投资金额；
   - 查看策略摘要；
   - 展开各项数据依据；
   - 标记执行状态；
   - 录入实际成交。

4. **历史记录页面**
   - 按时间查看历史策略；
   - 查看当时的数据和依据；
   - 比较建议与实际执行。

5. **候选池页面**
   - 主产品和备用产品；
   - 选择理由；
   - 当前状态；
   - 上次和下次复核时间；
   - 历史变更记录。

6. **设置页面**
   - 用户档案；
   - 风险和仓位约束；
   - 数据源状态；
   - 本地数据备份和恢复入口。

### 11.2 可视化

MVP 至少包含：

- 资金桶配置环形图或条形图；
- 当前比例与目标区间对比图；
- 账户和币种分布图；
- 组合历史净值或市值曲线；
- 单个指数水温的分项条形图；
- 最新策略执行前后配置对比。

图表必须服务于决策，不以增加图表数量为目标。所有图表同时提供数值和更新时间。

## 12. 回测模块

### 12.1 定位

回测作为独立模块实现，不进入每月核心流程。它用于生成可重复的策略分析报告，比较固定定投、固定配置和水温动态分配。

### 12.2 比较策略

- 每月固定金额定投；
- 固定目标比例并用新增资金再平衡；
- 基础配置加水温倾斜；
- 水温倾斜加现金/纯债防守。

### 12.3 主要输出

- 年化收益；
- 资金加权收益；
- 年化波动；
- 最大回撤和恢复时间；
- 滚动 3 年、5 年收益分布；
- 五年窗口达到目标收益率的比例；
- 最差年度和最差滚动 12 个月；
- 交易次数、费用和现金拖累；
- 相对固定定投基准的差异。

### 12.4 运行时机

- 第一版策略规则确定后运行一次正式报告；
- 规则、候选指数、费用假设或资金桶发生变化时重新运行；
- 后续可半年或一年例行更新；
- 不使用回测历史收益排名直接选择跟踪同一指数的基金。

## 13. 模块拆分、实现顺序与验收

### M0：项目基础与技术验证

范围：

- 确定本地运行方式；
- 验证 Codex 调用本地工具；
- 验证本地页面读取同一份记录；
- 定义配置、数据和报告目录；
- 建立测试框架和示例数据。

验收：

- Codex 能调用一个测试工具并获得结构化结果；
- 页面能读取并展示一份示例 JSON；
- 用户隐私数据不会进入 Git；
- 一条命令能够启动本地服务。

### M1：本地数据与用户档案

范围：

- 用户档案模型；
- 初始化字段；
- 读取、写入、校验和版本管理；
- 备份与恢复基础能力。

验收：

- 能保存和重新加载完整档案；
- 必填字段缺失时明确报错；
- 冲突约束能够被识别；
- 测试数据和真实数据隔离。

### M2：持仓账本

范围：

- 账户、标的、持仓和交易数据模型；
- 手动增删改；
- 多币种市值换算；
- 资金桶和集中度计算。

验收：

- 给定固定持仓样例，市值和占比计算正确；
- 编辑后能够持久化；
- 缺失价格时不伪造市值；
- 东财、港股通和 Revolut 能分别汇总。

### M3：市场数据刷新

范围：

- 股票、基金、指数、净值和汇率数据接口；
- 数据缓存；
- 时间、来源和失败状态；
- 手动刷新流程。

验收：

- 可用测试标的完成刷新；
- 每条数据有来源和时间；
- 单一数据源失败不破坏已有持仓；
- 过期数据触发明显提示。

### M4：候选池与基金筛选

范围：

- 指数席位；
- 主产品和备用产品；
- 硬性排除和评分；
- 候选池版本；
- 定期和事件复核。

验收：

- 同一输入产生可复现排序；
- 限购、高溢价和数据缺失产品能被排除；
- 每个主产品有选择理由；
- 候选池变更有历史记录。

### M5：水温与个股评分

范围：

- 指数水温子项和总分；
- 观察名单个股经营指标；
- 数据质量和红线；
- 评分版本管理。

验收：

- 固定数据得到固定分数；
- 子项能够解释总分；
- 未来数据不会进入历史时点评分；
- 数据不足时降低置信度或停止推荐。

### M6：资金分配与风险校验

范围：

- 目标配置区间；
- 新增资金分配；
- 现金保留；
- 交易和集中度限制；
- 卖出触发条件。

验收：

- 分配金额与输入资金一致；
- 不突破硬性仓位约束；
- 不生成无意义的小额操作；
- 所有候选不合格时能够全部保留现金；
- 风险校验失败时不输出正式建议。

### M7：Agent 工作流与策略记录

范围：

- 多轮初始化；
- 月度投资请求；
- 工具编排；
- 结构化策略输出；
- Markdown 报告；
- 执行状态和实际成交。

验收：

- Agent 能从自然语言提取金额和币种；
- 缺少关键信息时会追问；
- Agent 解释与结构化策略一致；
- 策略能落盘并重新读取；
- 模型不能绕过确定性风控。

### M8：本地页面与可视化

范围：

- 总览、持仓、策略、历史、候选池和设置页面；
- 手动刷新；
- 数据编辑；
- 必要图表；
- 加载、错误和空状态。

验收：

- M1～M7 的核心数据均可查看；
- 持仓可以安全编辑；
- 刷新状态和数据时间清楚可见；
- 图表与底层数值一致；
- 页面刷新后数据不丢失；
- 仅能从本机访问。

### M9：独立回测模块

范围：

- 数据准备；
- 基准策略；
- 动态策略；
- 费用和现金流模拟；
- 防前视和样本外验证；
- 报告生成。

验收：

- 固定输入可重复得到相同结果；
- 包含费用前后结果；
- 能与固定定投基准比较；
- 明确展示失败时期和最大回撤；
- 报告注明历史模拟不代表未来收益。

### M10：端到端验收

完整场景：

```text
初始化用户
→ 录入三类账户持仓
→ 手动刷新数据
→ 输入人民币和瑞典克朗新增资金
→ Agent 生成并解释策略
→ 页面展示策略前后配置
→ 保存报告
→ 录入实际成交
→ 页面更新新持仓
```

验收：

- 全流程无需手工修改底层数据文件；
- 中途退出后能够继续；
- 关键数据均可追溯；
- 不发生自动交易；
- 测试结果和已知样例一致。

## 14. 非功能要求

- **隐私**：持仓、档案、密钥和策略记录仅保存在本机，并默认忽略 Git。
- **可解释**：所有建议必须关联具体数据和规则，不只输出自然语言结论。
- **可复现**：相同数据和规则版本应得到相同核心分配结果。
- **可追溯**：保存数据、规则、候选池和策略版本。
- **失败安全**：数据缺失、过期或风控失败时停止正式推荐。
- **简单启动**：MVP 应提供统一的本地启动入口。
- **可测试**：每个模块提供单元测试、固定样例和模块验收测试。
- **可备份**：个人数据和策略记录可以导出及恢复。
- **性能**：在 M1 Mac 上，除外部数据请求和模型响应外，普通页面和本地计算应快速完成。

## 15. MVP 不包含

- 自动下单、自动转账、自动换汇；
- 券商持仓自动同步；
- 实时或分钟级行情；
- 每日自动交易；
- 全市场基金和个股扫描；
- 多用户、账号系统和公网部署；
- Revolut 资金汇回国内；
- 税务、法律和跨境合规结论；
- 收益保证；
- 原生手机 App；
- 面向公众的投资顾问服务。

## 16. 开发前必须确认的事项

以下事项不阻塞 M0～M2，但应在对应模块开发前确认：

1. Codex 与本地工具的最终集成方式；
2. 本地页面技术方案；
3. 第一批市场数据源；
4. 中国宽基最终选择；
5. 五类主产品和备用产品；
6. Revolut 实际可购买的 ETF 代码；
7. 初始目标配置区间和最大回撤；
8. 指数水温公式、权重和阈值；
9. 本地数据备份位置；
10. 外币资产统一展示的基准币种，默认建议使用人民币。

## 17. 第一版完成定义

只有同时满足以下条件，MVP 才算完成：

- M0～M10 验收通过；
- 用户能够通过 Agent 完成初始化和一次月度策略生成；
- 策略的金额、依据和风险均可查看；
- 策略及实际执行可以保存和恢复；
- 页面能够展示并编辑持仓；
- 手动刷新后能够展示最新可用价格或净值；
- 回测模块能够生成正式的基准比较报告；
- 没有自动交易或未经确认的持仓修改；
- 真实个人数据不进入版本库。
